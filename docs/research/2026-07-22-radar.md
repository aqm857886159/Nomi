## 🟢 可落地（这月能进 Nomi）

### Keyframe-Anchored Identity Preservation：给 image-first「多动作镜」一条 training-free 三段式——动作感知 prompt 改写→每动作出「终态关键帧」→关键帧间多参考+identity 引导补齐（2026-07-20，两天新）

[arxiv 2607.17985](https://arxiv.org/abs/2607.17985) · 作者 Zhenjie Liu / Binyan Chen / Hao Chen / Tong Pan / Shangfei Wang（USTC 王上飞组特征，未在页面逐一确认）· IPVG26 挑战赛 Track 2 官方榜第三 · 本轮**未核实到公开代码仓**（挑战赛方案，两天新，多半还没放码）· 日期：2026-07-20。

- **它治的正是 Nomi image-first 的一个真实开口**：IPVG26 把「身份保持」从「一句整体 prompt」升级成「**一串带时间戳的动作 caption，主体要按顺序做完这些动作、且全程身份不变**」。论文点破的失败模式——「end-to-end video generators are prone to appearance drift as motion accumulates and the depicted actions change」——**就是 Nomi「一镜一张首帧图→图转视频」在「这一镜里人物要连做好几个动作」时会踩的**：单张首帧只约束住起始画面，视频跑到后半段动作变了、人物就漂了。
- **三段式（全在编排/生成调度层，Nomi 三段各有对应件）**：
  1. **action-aware prompt polishment**——把「带时间戳的动作序列」改写成图像生成 prompt，每条**指定该动作的「终态」（terminal state）**。对应 Nomi：拆镜头规划师/AI 优化写图 prompt 时，多动作镜不写一张笼统首帧，而是**按动作切成「每动作一张终态关键帧」**的 prompt 序列。
  2. **identity-preserving generation**——逐帧出关键帧，每帧**同时条件在（参考身份 + 前一关键帧）**上，把「不随时间变的外观」和「随时间变的姿势」解耦。对应 Nomi：这串关键帧图就是给图转视频的**有序首帧组**（不是单帧），身份靠 Nomi 已有 `@imageN` 多参考底座锁。
  3. **identity-aware inference enhancement**——关键帧之间的过渡段用多参考引导 + identity 驱动的 noise searching 补齐。对应 Nomi：这一步碰采样（noise searching）Nomi 走 vendor 黑盒**接不了**，退化成「让 vendor 的图转视频在相邻两关键帧间插值」，靠首尾帧模型（Seedance 系）或多帧条件近似。
- **落到哪个文件 + 怎么改**：
  - `src/workbench/generationCanvas/agent/convertShotToVideo.ts`（已存在）——图转视频桥。加一条能力：镜的动作数 ≥2（或时长 > 阈值）时，不只传单首帧，而是**按动作终态传一组有序关键帧**（首尾帧模型直接用尾帧位；支持多帧条件的 vendor 传多帧）。
  - `src/workbench/generationCanvas/agent/storyboardPlan.ts`（328 行，`:285` 是 GroundShot 排序 backlog 锚点）+ AI 优化节点 prompt 模板——**主改动**：规划师给「一镜多动作」的镜写图 prompt 时，产出**每动作一张终态关键帧**的 prompt 序列，而不是一张笼统首帧。和上轮 ReBind 的 `@imageN` 显式归属**同一处改**（都在规划师写多参考/多帧 prompt 的地方），可并成一个 commit。
  - 身份锁复用 `src/workbench/assets/promptMentions.ts`（83 行，`@imageN` 单源投影）——关键帧序列里每张都绑同一 `@image1=主角`，与 ReBind 落地共用底座，不开第二套（P1）。
- **为什么进 🟢**：前两段（prompt 改写 + 终态关键帧 + 逐帧条件在前帧）**全在编排/生成调度层**，training-free、model-agnostic；Nomi 图转视频桥 + 多参考底座 + 首尾帧能力全现成，是「把首帧从单帧升成有序关键帧组」不是从零造。它拿这套在公开榜排到第三，说明思想有效不是纸面。
- **诚实边界**：① 第三段（noise searching）碰采样，Nomi 接不了，过渡段质量取决于所接 vendor 的多帧/首尾帧插值能力——**不支持多帧条件的 vendor 只能退回单首帧、吃不到这个收益**，按模型能力位 gate（`supportsReferenceImages` 同款思路，别硬上）；② 它是挑战赛方案、无公开代码，照三段式思想自实现，别 bluff 抄了它的实现；③ 「终态关键帧」拆得越细、图转视频要插值的跨度越大、中途越可能崩——切动作的粒度要 A/B 调，不是越细越好（与 KeyFrame-Compass 上轮结论「关键帧越密退化越狠」一致，两篇互印证）。

## 🟡 架构教训（影响 Nomi 设计决策，非即插即用）

### FilmWorld：整片级 agent 系统正式成型——「construction（落成有状态世界实体）+ evolution（闭环校验保因果一致）」几乎是 Nomi 架构的 1:1 学术镜像，且闭环校验就在编排层（2026-07-21，一天新）

[arxiv 2607.19038](https://arxiv.org/abs/2607.19038) · [项目页 filmworld-ai.github.io](https://filmworld-ai.github.io)（本轮实测 200，页面未见 GitHub 码链）· 作者 Jialong Zuo / Haotong Zuo / Shiwei Zhang / Xiang Wang / Chen Li / Nong Sang / Changxin Gao / Xiang Bai（**HUST 华科 Nong Sang/Changxin Gao/Xiang Bai + 阿里通义 Shiwei Zhang/Xiang Wang = Wan 系**，页面未列 affiliation，按作者群判）· 日期：2026-07-21（本轮窗内最新）。

- **它把 Nomi 正在搭的东西整套写成了论文**：novel→film 形式化成 **dynamic cinematic world modeling**，拆两相：**construction**——把「抽象、欠指定的文学语言」落成「具体、有状态、持久的世界实体」（world entity state modeling with **visual anchoring** + state-driven shot planning）；**evolution**——这些实体随剧情推进**动态更新**以维持跨场景因果一致（state-anchored visual generation + **cross-shot dynamic state propagation** + **closed-loop state verification**）。**对照 Nomi**：construction ≈ 拆镜头规划师（剧本→设定卡/世界实体→镜头计划），evolution ≈ 闭环回灌（生成→校验→把新状态传播进后续镜）。这不是又一篇「多 agent 拆镜头」——它把**「实体状态」当第一类对象贯穿全程 + 闭环校验维持因果**，正是 Nomi 命门（跨镜身份/连贯）的同解。
- **关键确认（WebFetch 项目页 + 摘要读得）**：closed-loop state verification 与 cross-shot state propagation **归 evolution-side agents、在编排层做，不在扩散模型内部**——这点决定了它对 Nomi**可借鉴不是可膜拜**：Nomi 走 vendor API，凡编排层的机制（状态建模/传播/校验）能抄，凡碰模型内部的不能。它整套恰好在编排层，与 GroundShot（实体调度）/ CoTriSyGen（三元组闭环）/ ReBind（参考归属）本轮合流成**同一条「编排层 decompose-stitch + 状态回灌」主干**——学界第十轮把这条主干从「零件」（调度、归属、记忆各一篇）收敛成「整机」（FilmWorld 一套端到端）。
- **影响 Nomi 哪个决策**：① **给 Nomi 架构一个最强同向背书 + 一张更完整的施工图**——Nomi 前九轮零散补的件（GroundShot 调度 / ReBind 归属 / KeyFrame 首帧忠实 / 闭环回灌）在 FilmWorld 里是一套系统的各模块，说明 Nomi 的方向是学界正在收敛的主干、不是孤注；②「**world entity state modeling with visual anchoring**」点出一个 Nomi 设定卡可强化的维度——设定卡不只存「角色长啥样」，还该存**「实体当前状态」并随剧情推进更新**（如「外套：第 3 镜被撕破 → 之后镜都该是破的」），对应 Nomi 记忆 `memory-system-redesign-2026-06-20` 的「结构化设定卡」方向再添一证：设定卡应是**可变视觉状态**（CoTriSyGen 上轮已提「mutable visual state」，FilmWorld 把它做进整片级系统）。近期不必大改，但拆镜头执行链路做大改时，「实体状态随镜演进 + 状态传播进后续 prompt」该是设计目标之一。
- **诚实边界**：① 未放代码（仅项目页），底层用哪个视频模型、是否 model-agnostic 摘要/页面未点名——「整套在编排层」是按 evolution-agent 描述判断，实现细节要等全文/放码核实（R5），别当落地源；② FilmEval 名字与既有 **FilMaster（2506.18899）的 FilmEval** 及另一 dialogue-to-film 工作的 FilmEval **撞名**，是各家自造同名 benchmark，别混为一谈、也别声称是同一考卷；③ 它的「outperform SOTA agent systems」是在自家 FilmEval 上测的，Nomi 不接它的模型、只吃架构思想与状态建模维度。

### GEST-Engine：「显式可检查世界模型 vs 学习潜变量」——事件图→游戏引擎确定性渲染，object permanence/时序一致「by construction」保证，正面背书 Nomi 节点画布=显式状态这条路（2026-07-14）

[arxiv 2607.12231](https://arxiv.org/abs/2607.12231) · 作者 Nicolae Cudlenco / Mihai Masala / Marius Leordeanu（罗马尼亚 Bucharest/IMAR 系，GEST 系列作者）· 是 [GEST(2604.10383)](https://arxiv.org/abs/2604.10383) 的完整技术报告 · 日期：2026-07-14。

- **核心主张 = Nomi 节点画布哲学的学术版**：与其把世界状态编码成「学习出来的 latent」，不如维持一个**完整、可检查（inspectable）的世界表示**——谁在、在哪、在做什么、拿着什么、事件时空关系如何——用形式化 **Graph of Events in Space and Time (GEST)** 表达，在商用游戏引擎里**确定性实现**。GEST 由「LLM Director 通过工具调用规划、每步被程序化 state backend 校验」产出，**every generated specification is executable by construction**（每条规格生成即可执行、不会非法）。四阶段执行（图解析校验→实体动作 grounding→Allen 时序约束用 Floyd-Warshall 传递闭包解→执行捕获），一遍出 RGB+深度+分割+骨架+关系图+时序映射，零边际标注成本。**因为每帧都能回溯到语义规格，object permanence/多主体协调/时序一致是 by construction 保证的、不靠模型自觉。**
- **影响 Nomi 哪个决策**：① **正面背书 Nomi「节点画布=显式、可检查、可版本化的创作状态」这条差异化路线**——记忆 `palmier-pro-deep-analysis` 里判过 Nomi 该占「关系图+跨镜身份+显式版本树」的空白，GEST-Engine 从学术侧给了同样的第一性论证：**显式可检查状态 > 黑盒 latent**，尤其在需要一致性/可追溯的严肃创作里；② **「LLM 规划经程序化 backend 校验、生成即合法」直接对应 Nomi MCP/agent 工具应「每步工具调用被校验、非法即挡」**——与记忆 `防呆评审`/`retry-must-not-wrap-paid-submit` 的防呆纪律同向，给 Nomi agent 工具设计一条原则：**agent 产出的每个画布操作/镜头规格都该被 store 侧校验器验过再执行，别信 LLM 自觉**（ReBind 上轮已在 prompt 层说过「通用 MLLM 写不可靠要校验兜」，GEST-Engine 把它上升成整个 agent 系统的设计律）；③ **反面确认 Nomi 的边界**——GEST-Engine 走「游戏引擎确定性渲染」出的是**素模/合成视频**（当训练数据/benchmark 用），不是成片像素；Nomi 的 3D 导演台走的正是这条（渲小片喂 vendor 全能参考），与它同源，但 Nomi 最终要 vendor 出真实像素、不停在引擎渲染——**别被它诱导去做「纯引擎出片」，那是数据生成不是创作成片**。
- **诚实边界**：它面向「造视频理解的训练数据/评测/诊断工具」，不是面向创作者出成片；Nomi 吃的是「显式可检查状态」哲学 + 「agent 每步被校验」原则，不吃它的引擎渲染管线（Nomi 不出素模成片）。

### ShotPlan：显式多镜规划 + 帧级转场（planning token + FRoPE）——又一篇印证「镜头转场要显式规划」，但机制在扩散模型内部、Nomi 只吃结论不吃实现（2026-07-20）

[arxiv 2607.17675](https://arxiv.org/abs/2607.17675) · 作者 Su Guo / Guangce Liu / Haosen Yang / Jiepeng Wang / Cong Liu / Junqi Liu 等 · 建在视频扩散基座上 · 日期：2026-07-20。

- **机制**：引入 **learnable planning tokens** 捕捉镜级转场线索，与原生视频 token 拼接来控制转场时间戳；planning token 配 **Fractional Temporal RoPE (FRoPE)**，让转场在**帧级**建模。报告显著超现有电影级视频生成方法、镜间一致性更强。
- **为什么只进 🟡**：核心（planning token 嵌进视频 token 序列 + 改 RoPE）**必须能碰模型内部**，Nomi 走 vendor 黑盒**接不了**，drop-in 不了。价值在**结论方向**——它和 FilmWorld/DirectorBench/KathaTrace 本轮再次合流坐实「**镜头转场是全行业中心瓶颈、且需要显式规划（不能指望模型自发）**」。对 Nomi 的信号：Nomi 拆镜头把「转场/镜序」显式化（时间轴 shot→shot 链 + 规划师排序）是对的方向，学界连模型侧都在往「显式 planning token」走，说明「显式转场规划」不是 Nomi 的权宜、是共识刚需；反过来也确认 Nomi 该继续在**编排层**把转场做显式（GroundShot 式排序 + 时间轴链），别指望 vendor 模型自己接好镜。
- **诚实边界**：训练进模型内部的方案，Nomi 一律接不了；本条只闭合方向 4（多镜叙事/转场），不产生落地项。与上轮 ReBind/本轮 FilmWorld 同属「快手/字节/通义系在把多镜能力下沉进模型」的产业信号（记忆 `browser-capture-trust-and-3d-keyframe-shipped` 里 P4 层「预留原生多镜能力位」的判断本轮再添一证）——某 vendor 放出「原生多镜+转场控制」API 时，Nomi 拆镜头执行器该能整段委托，别把「一镜=一次调用」焊死。

## 🔵 对标基准（接进 eval 体系自评 Nomi）

### MultiRef-Compass：多参考→音视频生成的评测尺子——正补上轮 ReBind「多参考绑定对不对」缺的量化轴（multi-entity binding + reference consistency）（2026-07-15）

[arxiv 2607.14189](https://arxiv.org/abs/2607.14189) · 作者 Xiaohan Zhang / Yuqing Wen / Junlin Chen / Yuqi Tang / Yiting He / Lizhuo Shao 等 · 350 样本 / 8 个代表 MR2AV 系统实测 · 本轮未核实到公开代码仓 · 日期：2026-07-15。

- **它量的正是上轮 ReBind 落地后该验的东西**：MR2AV（multi-reference-to-audio-video）= 多张参考 + 文本指令 → 生成，模型必须「**不仅忠实保留每张参考，还要正确 bind 并 compose 多个被指代实体**成连贯视听事件」——**这就是 Nomi「三张参考图分不清谁是谁→张冠李戴」痛点的正式任务定义**。上轮 ReBind 给了解法（`@imageN` 显式归属）、Nomi 已在落，但「归属改了之后到底绑对了没」之前没有尺子。MultiRef-Compass 的四维协议里，**Reference Consistency（每张参考保真度）+ 隐含的 multi-entity binding（多实体绑对没）** 两维直接量这个。14 子指标 + rejudging-enhanced MLLM-as-Judge。
- **怎么接进 eval 体系**：① 给 Nomi **参考一致性 eval lane** 加两条轴——「每个参考实体在成片里保真度」+「多实体是否绑对（没张冠李戴）」，judge 走现有 VLM 链路（Nomi VLM judge 基建现成，成本在 prompt 工程不在训练）；② 这是 ReBind 落地的**A/B 验收尺**：改 prompt 归属（`promptMentions.ts` 的 `@imageN`）前后各跑一遍，量张冠李戴率降没降——上轮 ReBind 条目自己标了「落地后要用 Nomi eval A/B 验证，别默认等效」，MultiRef-Compass 正好是那把尺；③ 它的「asset-composition pipeline（可控合成多参考样本）」思路可借来给 Nomi 造参考一致性测试集（human-object-scene composition 三类），不用等它放数据。
- **诚实边界**：① 无公开代码/数据核实到，四维协议要照定义自实现（Nomi 已有 VLM judge 基建）；② 它含音视频联合（audio-visual consistency 维），Nomi 现阶段 BGM/配音是后期合、不是联合生成，那一维**不适用**、只取 Reference Consistency + binding 两维；③ 其 rejudging-enhanced judge 的可靠性与 Nomi 域未必同构——与上轮 JudgeFit 结论合读：**换 judge 模型要重校 rubric**，接进来时至少重跑一次标注对齐。

### Learning Explicit Physical Parameter Control（PhyParam）：NEWTON「规格瓶颈」线的落地版 + PhyParam-Bench——「能算的参数别用形容词」再添物理域实证，但模型内部、Nomi 只吃 benchmark 视角（2026-07-21）

[arxiv 2607.18924](https://arxiv.org/abs/2607.18924) · 作者 Yanxun Li / Hao Wen / Bingze Song / Jiashu Zhu / Aiming Hao / Chubin Chen / Jintao Chen / Jiahong Wu · PhyParam-Dataset 13 万物理模拟视频 + PhyParam-Bench + 承诺放数据/码 · 日期：2026-07-21。

- **它坐实的正是上轮 NEWTON 那条**：现有 i2v「生成看似合理的运动，但不被显式物理因由可靠支配，且实例级约束在多物体交互里会泄漏/纠缠」——归因于两个缺失：**大规模细粒度物理参数化** + **把物理属性正确绑到实例**的模型设计。解法 PhyParam：把 object-level 力/质量/摩擦/回弹 + scene-level 重力**当显式条件**喂 i2v（轻量 physical-attention routing）。核心呼应 NEWTON 的「specification bottleneck」——**决定动力学的参数得显式给，别指望模型从形容词补出来**。
- **怎么接进 eval 体系 / 影响 Nomi**：① **PhyParam-Bench 是「物理一致性」这条 lane 的现成考卷**——Nomi 若要量「生成的运动物理上合不合理」（多物体碰撞/下落/摩擦），可参考它的多级协议（temporal dynamics / spatial stability / semantic-physical alignment）；但 Nomi 现阶段主战场是叙事/身份/构图，物理一致性优先级低，**当 backlog 里的一条候选尺、非近期必接**；② 第一性上给 Nomi 3D 导演台→prompt 链路再添一证（与 NEWTON 同向）：**焦段 mm/机位坐标/物体位置等能从 3D 台/结构化字段算出来的，都别退化成「缓慢/近一些/用力」这类形容词**——审视拆镜头规划师 prompt 模板里的可量化形容词是持续动作项（上轮 NEWTON 已记，本轮 PhyParam 加固，不重复落点）。
- **诚实边界**：① PhyParam 本体是**训练的 i2v 模型**（碰模型内部 + 需 13 万模拟数据训练），Nomi **接不了模型本体**，只吃「显式参数化」原则 + benchmark 视角；② 物理域，与 Nomi 主战场（叙事/身份）相关度中等，故进 🔵 尾条而非头条；③ 承诺放数据/码但本轮未放出，接 benchmark 前等真放出（R5）。
